# CurioSync Concept  v0.1

> 本文档记录产品从概念到 MVP 阶段的核心思考过程，供后续开发与决策参考。\
> 最后更新：2025年初

## 一、产品起点与核心洞察

### 我们在解决什么问题

现有 AI 聊天工具（ChatGPT、Claude 等）在学习场景下存在一个根本性缺陷：

```javascript
它们是优秀的问答机器  
但不是学习伙伴  

用户得到了答案，但没有发生真正的学习  
知识是碎片化的，没有沉淀  
对话结束后，什么都没留下  
```

更深层的问题是：**人类的好奇心是发散的，但学习需要一定的结构**。两者之间存在天然张力，现有工具完全倒向了发散一侧。

### 核心洞察

> **好的学习不是压制好奇心，而是引导好奇心形成结构**

CurioSync 不试图让用户"按计划学习"，而是在用户自由探索的过程中，悄悄地帮他们建立知识结构。

---

## 二、产品定义

### 产品名称

**CurioSync**

- Curio = Curiosity（好奇心）
- Sync = 同步、整合
- 含义：让好奇心与知识结构同步生长

### 一句话定义

> 一个以对话为载体、以好奇心为驱动、能够自动沉淀学习成果的 AI 学习伙伴

### 产品的本质区别

```javascript
普通 AI 对话工具：  
  用户问 → AI 答 → 结束  

CurioSync：  
  用户探索 → AI 引导 → 知识沉淀 → 地图生长 → 更好的探索  
```

## 三、核心产品机制

### 3.1 两种学习模式

#### 兴趣模式（默认）

- 用户自由探索，无需预设目标
- 系统在后台追踪话题走向
- 在合适时机做轻量引导，不打断流动
- 探索成果自动沉淀到知识地图

#### 目标模式

- 需要有明确的学习目标才能进入
- 系统在自由探索的基础上增加路径意识
- 偏离目标时记录兴趣点，适时拉回
- 目标本身是动态生长的，不是静态大纲

#### 两种模式的关系

```javascript
不是平行关系，而是生长关系：  

兴趣探索 → 目标自然生长 → 目标驱动 → 新兴趣 → 目标扩展 → ...  
```

**关键设计**：两种模式之间切换是无损的，目标挂起但不消失，掌握度和书签完整保留。

---

### 3.2 对话调控机制（三种策略）

这是 CurioSync 区别于普通 AI 对话工具的核心能力。

#### 策略一：收敛聚焦

```javascript
触发条件：话题链过长，发散程度过高  
介入方式：在完成当前回答后，生成话题小结  
           提示用户选择继续发散还是收敛整理  
目的：防止探索变成漫无目的的漂流  
```

#### 策略二：路径拉回（目标模式）

```javascript
触发条件：对话偏离学习目标  
介入方式：完成当前回答 → 记录兴趣书签 → 提示回归路径  
特别设计：不强制拉回，给用户选择权  
           偏离的内容不丢失，记入兴趣书签  
```

#### 策略三：阶段跃迁

```javascript
触发条件：当前话题探索相对充分，出现自然收敛点  
介入方式：总结当前阶段 → 提供 2-3 个下一步方向选项  
目的：在对话自然流动中推动学习进展  
```

#### 介入的基本原则

- **永远先完成当前回答**，再介入
- 介入以卡片/提示形式嵌入对话流，不弹窗
- 给用户选择权，不强制
- 任何介入都不丢失用户的探索内容

---

### 3.3 学习目标的生长机制

#### 目标不是前置条件，而是从探索中生长出来的

```javascript
来源一：用户主动构建  
  用户明确提出建立学习目标  
  系统先评估用户当前状态  
  生成"刚好够用"的目标，而不是完整大纲  

来源二：智能体适时建议  
  触发条件：对话有一定深度 + 话题有内聚性 + 用户持续感兴趣  
  建议方式：轻量插入，不打断对话  
  扩展原则：每次只扩展 1-2 个相邻方向，严格控制梯度  
```

#### 关键原则：梯度控制

```javascript
错误：用户问了一个问题，系统建议建立一个完整学科学习计划  
正确：用户探索了几轮，系统建议把这个方向作为小目标，  
      提出 1-2 个可以继续深入的方向  
```

---

### 3.4 知识图谱体系

#### 三层结构

```javascript
Layer 3：用户个性化层（最高优先级）  
  用户上传的课本、笔记、试卷  
  LLM 解析整理，转化为知识节点  

Layer 2：动态生成层  
  LLM 基于对话动态生成，结合搜索  
  标注来源和置信度  

Layer 1：预制基础层  
  官方/社区预制的学科知识骨架  
  适合应试类、有明确大纲的学习目标  
```

#### 知识图谱的特性

- 从学习过程中自动生长，不需要手动构建
- 只读，但有交互（点击节点查看关联内容）
- 节点状态反映掌握度（颜色深浅、大小）
- 与当前对话实时联动（正在探索的节点高亮）

---

### 3.5 掌握度体系

#### 掌握度的五个状态

```javascript
UNAWARE    → 未接触  
EXPOSED    → 有印象（听说过，说不清楚）  
UNDERSTOOD → 理解（能用自己的话解释）  
FAMILIAR   → 熟悉（能在新情境中运用）  
MASTERED   → 掌握（能举一反三，能教别人）  
```

#### 采集方式

**被动推断**（始终运行，用户无感知）

```javascript
用户提问方式 → 推断理解层次  
用户回应质量 → 推断掌握程度  
对话流畅程度 → 辅助判断  
用户表述方式 → 自己的话 vs 照搬术语  
```

**主动评估**（嵌入对话，按侵入程度分级）

```javascript
最低侵入：探究式追问（"那你觉得...呢？"）  
中等侵入：费曼式评估（"来教我一下X"）  
高侵入：项目式实践（真实任务中的表现）  
最高侵入：应试式测验（按需触发，克制使用）  
```

#### 核心设计原则

> **评估过程就是学习过程，学习过程就是评估过程**

评估不是单独的模块，而是自然嵌入在对话流中。唯一的例外是应试场景（不得不兼容现实）。

## 四、产品信息架构

```javascript
CurioSync  
│  
├── 主题 Topic（类似"工作区"）  
│     ├── 知识图谱（主题级别，所有对话共享）  
│     ├── 掌握度地图  
│     ├── 兴趣书签  
│     └── 对话列表  
│           ├── 正式对话（沉淀到知识图谱）  
│           └── 临时对话（不沉淀，随时开随时关）  
│  
└── 其他主题...  
```

**多对话的关键设计**：

```javascript
对话之间：上下文独立（对话历史不互通）  
图谱层面：共享同一个知识状态  

效果：用户可以从不同角度探索  
      每次探索的收获都沉淀到共享的知识地图  
```

## 五、多智能体架构思路

### 按时间尺度划分职责

```javascript
短时间尺度 → 对话智能体（关注当前这一轮对话）  
中时间尺度 → 观察智能体（关注当前对话的走向）  
长时间尺度 → 图谱智能体（关注整个主题的知识状态）  
```

### 三层智能体职责

#### 对话智能体 Dialogue Agent（每个对话一个）

```javascript
✅ 回答问题、讲解概念  
✅ 执行观察智能体下达的介入策略  
✅ 维护对话的流动感  
❌ 不判断何时介入  
❌ 不修改知识图谱  
```

#### 观察智能体 Observer Agent（每个对话一个）

```javascript
✅ 追踪对话状态（话题链、漂移程度、兴趣信号）  
✅ 从第三方视角判断介入时机  
✅ 选择介入策略，指令对话智能体执行  
✅ 向图谱智能体上报信号  
❌ 不直接与用户对话  
❌ 不直接修改知识图谱  
```

#### 图谱智能体 Knowledge Agent（每个主题一个）

```javascript
✅ 维护整个主题的知识图谱  
✅ 整合来自多个观察智能体的上报  
✅ 更新掌握度记录  
✅ 识别跨对话关联  
✅ 判断是否建议扩展学习目标  
❌ 不干预具体对话内容  
```

### 多对话不会"打架"的原因

```javascript
观察智能体A  只管对话A，向图谱智能体上报"信号"  
观察智能体B  只管对话B，向图谱智能体上报"信号"  
图谱智能体   收到所有信号，做综合判断  

关键：观察智能体上报的是信号，不是决策  
      决策权在图谱智能体，唯一，不冲突  
```

### 后期可扩展的智能体角色

```javascript
同学智能体   → 费曼学习法的对话对象（MVP 阶段作为对话智能体的角色模式）  
规划智能体   → 学习路径规划（MVP 阶段合并在图谱智能体中）  
资料智能体   → 处理用户上传文件（工具型，按需加入）  
```

## 六、技术栈决策

### 选定方案

```javascript
前端        Next.js + React Flow  
后端        FastAPI（Python）  
LLM 编排   LangChain → 后期迁移 LangGraph（同一家，可平滑升级）  
数据库      Supabase（PostgreSQL + pgvector）  
LLM         MiniMax（MVP）→ 后期多模型扩展  
部署        本地开发 → Vercel（前端）+ Railway（后端）  
```

### 关键架构原则

```javascript
API 优先设计  
  后端从第一天起按"提供 API 服务"设计  
  前端只是 API 的一个客户端  
  未来 iOS / Android 直接复用同一套后端  
  不需要重新开发  

全平台扩展路径：  
  MVP → Web（Next.js）  
  第二阶段 → iOS / Android（复用后端 API）  
```

### 成本估算

```javascript
本地开发阶段：仅 LLM API 调用费用（极低）  
云端部署后：100-200 元/月（MVP 规模）  
远低于 1000 元/月的约束  
```

### 开发方式

```javascript
规格文档驱动开发（推荐）  
  先完成完整技术规格文档  
  再交给 Claude Code 按规格实现  
  避免碎片化开发导致的结构混乱  

角色分工：  
  你  →  产品设计者 + 架构决策者  
  Claude Code  →  执行明确的功能实现  
```

## 九、MVP 概述

> 详细开发文档见《CurioSync MVP 开发规格文档》

### MVP 的核心目标

让用户第一次使用就能感受到：

> **"我在被引导着，真的在学一样东西"**\
> 而不是"我在用一个更聪明的搜索引擎"

### MVP 包含的能力边界

**做**

```javascript
✅ 主题创建与管理  
✅ 主题下多对话（含临时对话）  
✅ 兴趣模式下的对话流  
✅ 三种对话调控策略（收敛 / 跃迁 / 书签记录）  
✅ 知识图谱自动生长（从对话提取节点）  
✅ 知识图谱基础可视化（节点 + 连线 + 掌握度状态）  
✅ 兴趣书签  
✅ 被动掌握度推断（3档状态）  
```

**不做**

```javascript
❌ 目标模式（第二阶段）  
❌ 用户上传资料  
❌ 预制知识图谱  
❌ 费曼式 / 应试式评估  
❌ 跨对话关联识别  
❌ 移动端  
❌ 多模型切换  
```

### MVP 的智能体实现策略

```javascript
不实现真正的多智能体架构  
用单一 LLM + 结构化 Prompt 模拟三层行为  

验证核心体验后，再拆分为真正的多智能体  
避免过早的架构复杂性影响产品验证速度  
```
